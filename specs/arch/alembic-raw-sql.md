# ADR: Alembic with raw SQL, no ORM

> **Status:** Accepted
> **Date:** 2023-11-01 (Phase 1)
> **Scope:** All database schema management across every package

---

## Context and Problem Statement

Ze needs a migration strategy. The two obvious approaches are: use SQLAlchemy (ORM or
Core) to autogenerate migrations from model definitions, or write raw SQL migrations
managed by Alembic. The codebase uses asyncpg for runtime, which has no SQLAlchemy
integration.

---

## Decision Drivers

- asyncpg is the runtime driver; SQLAlchemy ORM does not support asyncpg natively
  without `asyncpg-compat` shims
- Domain types are dataclasses, not ORM models — there are no model definitions to
  autogenerate from
- Migration ownership is per-package (each package in `core/` owns its own migration
  chain); autogeneration would require a shared metadata registry
- Schema changes must be explicit and reviewable — autogeneration diffs can silently
  miss constraints or generate unsafe operations on live tables

---

## Considered Options

1. **SQLAlchemy ORM + Alembic autogenerate** — models drive schema, Alembic generates diffs
2. **SQLAlchemy Core + Alembic** — no ORM, but SQLAlchemy Table definitions as schema source
3. **Alembic raw SQL, no SQLAlchemy** — explicit `op.execute(...)` with hand-written SQL

---

## Decision Outcome

**Chosen option: Alembic raw SQL (Option 3).**

Migrations are hand-written SQL in `op.execute()` blocks. Each package owns its own
Alembic chain with a distinct prefix (`zc`, `zm`, `zcal`, etc.). `ze_api/migrate.py`
is a meta-runner that discovers all chains via `version_locations` and applies them
against a single `alembic_version` table.

### Positive Consequences

- Every schema change is explicit, reviewable, and unambiguous
- No autogeneration surprises (unexpected `DROP COLUMN`, missing index, wrong type)
- No SQLAlchemy dependency in domain code — asyncpg pools used directly everywhere
- Per-package migration chains make ownership clear and enable `depends_on` for
  cross-package ordering

### Negative Consequences / Trade-offs

- More verbose than autogenerated migrations — every column must be typed manually
- No model-based query building — all queries are raw SQL strings or asyncpg templates
- Autogenerate safety checks (detecting drift) are unavailable — the schema can drift
  from the codebase if a migration is missed

---

## Pros and Cons of the Options

### Option 1 — SQLAlchemy ORM + autogenerate

**Cons:** Requires ORM models (Ze has dataclasses); asyncpg is not natively supported;
autogenerate diffs are non-deterministic with multiple packages.

### Option 2 — SQLAlchemy Core + Alembic

**Pros:** Table definitions as schema source, no ORM overhead.

**Cons:** SQLAlchemy dependency in every package that defines tables; async support
still requires shims; adds a layer between the SQL and the reader.

### Option 3 — Alembic raw SQL

**Pros:** Explicit, auditable, no extra dependencies. SQL is what the DB actually runs.

**Cons:** Verbose; no autogenerate drift detection.
