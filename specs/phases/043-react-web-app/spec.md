## Purpose

This phase delivers the React web client that replaces `ze-app` (Flutter) as Ze's primary
interface. The app connects to the FastAPI backend over WebSocket, renders agent responses
as React components (using component descriptors from Phase 41), and runs in the browser
with zero compilation or platform friction.

The app is single-user and self-hosted. There is no sign-up, no account, no distribution
requirement. The user opens it in any browser and configures a server URL and API key.
A Tauri desktop wrapper can be added later without any changes to the React code.

---

## Responsibilities

- Maintain a persistent WebSocket connection to the backend, with automatic reconnect.
- Render inbound `message` frames as a chat timeline — text (markdown) + zero or more
  components below it, in order.
- Render all 8 component types as React components using TypeScript types generated from
  `scripts/generate_components.py`.
- Send user messages and `ack` frames over the WebSocket.
- Load message history via `GET /api/messages` on app open / reconnect.
- Show a "Ze is thinking…" state (disable send, show typing indicator) while a response
  is in flight.
- Persist the server URL and API key in `localStorage`.
- Refresh data screens when `refresh` frames arrive from the backend.

---

## Out of Scope

- Desktop distribution (Tauri wrapper deferred to a later phase).
- Multi-user or multi-account support.
- Voice input from the browser (the backend handles audio; this is a future concern).
- Image capture (same — backend handles it).
- Component interaction callbacks beyond confirm tap — deferred.
- Offline mode — messages are loaded from the backend; no local cache beyond the session.
- Deep link URL schemes — browser URL routing replaces the `ze-app://` scheme.
- PWA / service worker — deferred.

## Design System

Ze's visual language is a dark cosmic field: pure black canvas, a single saturated violet
as the authority color, and white type that glows against the void. Inspired by
[Aceternity UI](https://ui.aceternity.com) for animated primitives and the Dala design
system for token and layout discipline.

**Rules:**
- Black (#000000) is the only background surface. No nested panels, no card elevation.
- Plum Voltage (#8052ff) is the only filled chromatic color — one button, one accent.
- Depth comes from color contrast and negative space, never from shadows or gradients.
- All interactive surfaces use 24px border-radius (pill for buttons, rounded square for cards).
- No shadows, no gradients, no textures.

### Color Tokens

```css
/* globals.css */
@theme {
  --color-void:          #000000;  /* page background */
  --color-bone:          #ffffff;  /* primary text, borders, icons */
  --color-ash:           #bdbdbd;  /* secondary muted text */
  --color-smoke:         #9a9a9a;  /* tertiary text, nav resting state */
  --color-plum-voltage:  #8052ff;  /* primary action, accent — only chromatic fill */
  --color-amber-spark:   #ffb829;  /* secondary outlined accent */
  --color-lichen:        #15846e;  /* decorative icon accent */
}
```

### Typography

Inter is the sole typeface (substitute for Acronym). Weight and tracking carry all
hierarchy — no serifs, no display variants.

```css
@theme {
  --font-sans: 'Inter', ui-sans-serif, system-ui, sans-serif;

  /* Scale */
  --text-caption:    12px;  /* tracking: 0.05em,  lh: 1.5  */
  --text-body-sm:    14px;  /* tracking: 0.05em,  lh: 1.5  */
  --text-body:       15px;  /* tracking: 0.025em, lh: 1.5  */
  --text-subheading: 18px;  /* tracking: 0.025em, lh: 1.5  */
  --text-heading-sm: 24px;  /* tracking: 0.021em, lh: 1.3  */
  --text-heading:    36px;  /* tracking: 0.021em, lh: 1.2  */
  --text-heading-lg: 48px;  /* tracking: -0.04em, lh: 1.1  */
  --text-display:    78px;  /* tracking: -0.04em, lh: 0.9  */
}
```

Display text (48px+): weight 200, negative tracking — "etched in light" feel.
Body and UI text: weight 400–600, positive tracking for legibility on black.
Eyebrows/kickers: weight 600, uppercase, 0.05em tracking.

## WebSocket Client

```ts
// src/ws/useWebSocket.ts

export function useWebSocket(config: AppConfig) {
  // Returns:
  //   frames: ReadableStream<InboundFrame> (via Zustand store)
  //   send(frame: OutboundFrame): void
  //   isConnected: boolean
}
```

### Reconnect strategy

On disconnect: exponential backoff starting at 1 s, capped at 30 s.

On reconnect: server pushes `list_unread()` frames immediately. REST history load runs in
parallel. Messages are deduplicated by ID in a `Map<string, Message>` — frames from both
sources merge safely.

### Frame protocol

```ts
// src/ws/protocol.ts

// Server → Client
type InboundFrame =
  | { type: 'message';         message: Message }
  | { type: 'edit';            id: string; text?: string; components: ComponentDescriptor[] }
  | { type: 'confirm_request'; id: string; prompt: string; actions: ConfirmAction[] }
  | { type: 'confirm_cancel';  id: string }
  | { type: 'typing' }
  | { type: 'error';           detail: string }
  | { type: 'refresh';         screen: string }
  | { type: 'pong' };

// Client → Server
type OutboundFrame =
  | { type: 'message'; text: string; thread_id?: string; context?: ScreenContext }
  | { type: 'ack';     ids: string[] }
  | { type: 'command'; name: 'cancel' | 'costs' | 'memory' | 'contacts' }
  | { type: 'ping' };

type ScreenContext = {
  screen: string;
  goal_id?: string;
};
```

## Message Rendering

### `MessageBubble`

Each message renders as a bubble containing:
1. Text (if non-null), rendered as markdown via `react-markdown`.
2. Each component in order, rendered by `componentRenderer()`.
3. Timestamp.

User messages align right, assistant messages align left.

### `componentRenderer()` factory

```tsx
// src/components/ComponentRenderer.tsx

export function componentRenderer(descriptor: ComponentDescriptor): JSX.Element {
  try {
    switch (descriptor.type) {
      case 'table':    return <TableComponent    data={descriptor} />;
      case 'metric':   return <MetricComponent   data={descriptor} />;
      case 'list':     return <ListComponent     data={descriptor} />;
      case 'timeline': return <TimelineComponent data={descriptor} />;
      case 'progress': return <ProgressComponent data={descriptor} />;
      case 'confirm':  return <ConfirmComponent  data={descriptor} />;
      case 'form':     return <FormComponent     data={descriptor} />;
      case 'card':     return <CardComponent     data={descriptor} />;
    }
  } catch {
    return <></>;
  }
}
```

Unknown component types are silently skipped. New components added to the backend before
the app is updated degrade gracefully.

### Component specs

All components sit on the void (#000000). No shadows, no fills — structure comes from
hairline borders (1px bone at ~15% alpha) and color contrast.

| Component | Renders as |
|-----------|-----------|
| `TableComponent` | `<table>` in a scrollable `div` (`max-h-72 overflow-auto`); hairline grid lines in ash; header row bone weight 600; no alternating row fills |
| `MetricComponent` | Large value in bone weight 200 at 48px; label in smoke at 14px below; trend chip as pill with amber-spark border if present |
| `ListComponent` | List of items; hairline separator between rows; title in bone weight 600; subtext in smoke; status as pill with plum-voltage border |
| `TimelineComponent` | Vertical line in ash (1px); time label in smoke left; event card right with 1px bone border at 15% alpha; 24px radius |
| `ProgressComponent` | Vertical stepper; done = filled plum-voltage circle; active = pulsing plum-voltage ring (Framer Motion pulse); pending = outline circle in smoke |
| `ConfirmComponent` | 1px bone border card (24px radius) on void; prompt in bone; button row: primary = plum-voltage fill, secondary = 1px bone outline, danger = 1px amber-spark outline |
| `FormComponent` | Input fields: 1px bone border at 20% alpha, 24px radius, bone text on void, focus ring in plum-voltage; submit = plum-voltage fill pill |
| `CardComponent` | 24px radius card; left border 3px solid: info=plum-voltage, warning=amber-spark, success=lichen, error=red-500; body in bone; no fill behind it |

### `ConfirmComponent` tap behaviour

Tapping an action sends a `message` frame with `text = action.value`. The `ConfirmComponent`
disables itself after the first tap to prevent double-submission.

## Contextual Overlay

On non-chat screens a floating widget lets the user talk to Ze without leaving the current
screen. `Cmd+K` / `Ctrl+K` opens it from any screen.

**Mobile** renders a bottom sheet that slides up from the bottom edge.  
**Desktop** renders a small floating panel (368 × 520 px) anchored to the bottom-right
corner. The panel is draggable by its header and collapses to a round FAB when closed.
The FAB is hidden on the `/` chat page since the full chat is already visible there.

### Behaviour

- **Open:** FAB tap, `Cmd+K`, or programmatic `openFor(screen)` / `openForExecution(params)`.
- **Context:** every sent message carries `context: { screen, goal_id?, workflow_id?, execution_id? }` so Ze knows which page and entity the user is viewing.
- **Sessions:** the overlay uses its own persistent thread ID (separate from the main chat), stored in `localStorage` under `ze_overlay_thread_id`. Conversations are saved to the backend and appear in the Chat session history.
- **Open in chat:** the `↗` button in the header selects the overlay thread in the session store, navigates to `/`, and closes the widget — the full conversation opens in the main chat view.
- **Collapse (desktop):** the `⌄` button collapses the panel back to the FAB. The session is preserved.
- **Dismiss (mobile):** tap the backdrop, press `Esc`, or tap `×`.
- **Live update:** `refresh` frames from the backend trigger `invalidateQueries()` on the relevant screen's data while the overlay stays open.
- **Follow-up:** overlay stays open after Ze responds until explicitly dismissed.
- **Thinking state:** input disabled + typing indicator while Ze processes.

### Screen context values

| Screen | `context` sent |
|--------|---------------|
| Goals (list) | `{ screen: "goals" }` |
| Goals (goal focused) | `{ screen: "goals", goal_id: "<uuid>" }` |
| News | `{ screen: "news" }` |
| Reminders | `{ screen: "reminders" }` |
| Contacts | `{ screen: "contacts" }` |
| Costs | `{ screen: "costs" }` |

### `refresh` frame handling

```ts
// src/navigation/RefreshHandler.tsx

function handleRefreshFrame(screen: string, queryClient: QueryClient) {
  const keyMap: Record<string, string[]> = {
    goals:     ['goals'],
    reminders: ['reminders'],
    contacts:  ['contacts'],
    costs:     ['costs'],
    news:      ['news'],
  };
  const key = keyMap[screen];
  if (key) queryClient.invalidateQueries({ queryKey: key });
}
```

## Message History Load

On app open (or after settings change):

```
1. Connect WebSocket → server pushes list_unread() immediately
2. In parallel: GET /api/messages?since=<7 days ago>&limit=200
3. Deduplicate by ID in a Map<string, Message>
4. Render messages in ascending order, scroll to bottom
5. Send { type: "ack", ids: [...] } for all unread assistant message IDs
```

A "Load earlier messages" button at the top fetches the preceding 7-day window on demand.
